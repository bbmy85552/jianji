## 2. 背景与目标

当前项目已经完成基于 Stitch 风格生成的前端 UI 原型，工程形态为 Vite + React + TypeScript + Tailwind CSS 的单页应用。现有代码主要包含工作台、知识库和数据表三个静态视图，尚未实现后端服务、数据库、鉴权、文档编辑器、邮件、日历、协作共享和管理后台逻辑。

本产品目标是将“简记”建设为一个开源、自托管、支持多设备访问的文档中心。产品应具备知识文档编辑、结构化表格管理、任务工作台、邮箱聚合、日历提醒、用户与权限系统、多人共享协作和后台管理能力，同时保持部署成本可控，适合个人、团队和开源社区自行部署使用。

## 3. 产品定位

简记是一个轻量但完整的开源文档协作平台。它不追求完全复刻大型商业套件复杂度，而是参考语雀的知识组织体验、腾讯文档的共享协作体验、现代协作文档的现代编辑体验，以及结构化数据表的数据视图能力，形成适合自托管场景的文档中心。

核心定位如下：

- 面向个人和小团队的统一知识工作台。
- 支持富文本、表格、任务、邮件和日历的聚合工作流。
- 支持开源自托管部署，可在电脑、平板、手机等设备通过访问地址使用。
- 支持基础实时协作、分享链接和细粒度权限控制。
- 提供管理后台，便于管理员维护用户、用户组和系统配置。

## 5. 用户角色

### 5.1 访客

未登录用户，只能访问登录、注册、找回密码页面，以及公开分享且允许匿名访问的文档页面。

### 5.2 普通用户

登录后可创建、编辑、收藏、共享文档，使用数据表、工作台、邮箱、日历和个人设置。

### 5.3 文档所有者

拥有文档或空间的最高权限，可管理协作者、设置分享链接、调整用户权限、转移所有权、删除或归档文档。

### 5.4 协作者

通过邀请链接或指定邮箱加入协作，根据权限可查看或编辑文档，并可在协作界面看到其他协作者的在线状态和编辑进度。

### 5.5 管理员

可进入管理后台，管理用户、用户组、角色权限、系统邮箱配置、验证码策略、系统审计和部署参数。

## 7. 核心功能需求

### 7.1 工作台模块

工作台应以仪表盘形式展示用户当天最重要的工作，不展示会议纪要、项目计划、个人百科等默认卡片。

功能要求：

- 展示今日待办列表。
- 每条待办支持勾选完成、取消完成、编辑标题、设置日期、删除。
- 展示今日任务完成进度，包括完成数量、总数量、完成百分比。
- 提供快捷按钮添加待办事项。
- 支持选择任意日期并为该日期添加待办事项。
- 支持从文档、数据表和日历中汇总与今日相关的事项。
- 支持空状态：当日无任务时展示引导创建按钮。
- 支持移动端卡片化展示。

验收标准：

- 用户可添加今天或未来某一天的待办。
- 勾选待办后，进度条和完成百分比实时更新。
- 刷新页面后，待办状态保持不丢失。

### 7.2 文档中心模块

文档中心参考语雀的知识库组织方式和腾讯文档/现代协作文档的编辑协作方式。

功能要求：

- 支持空间、知识库、文件夹、文档四级组织结构。
- 支持新建文档、重命名、移动、复制、收藏、归档、删除。
- 支持文档树搜索，搜索结果显示标题、路径、更新时间。
- 支持最近编辑、共享给我、我的收藏。
- 支持 Markdown 快捷输入。
- 支持自动保存和保存状态提示。
- 支持历史版本查看和恢复。
- 支持文档导出为 Markdown、HTML、PDF。
- 支持从 Markdown 导入文档。

验收标准：

- 用户可创建文档并在文档树中看到结果。
- 文档内容可编辑、保存、刷新后恢复。
- 用户可通过搜索找到目标文档。

### 7.3 文档编辑器

编辑器应参考 Word 的常用编辑能力，同时结合现代块编辑体验。

基础编辑工具：

- 撤销、重做。
- 标题 1 至标题 6。
- 正文、引用、代码块、分割线。
- 加粗、斜体、下划线、删除线。
- 上标、下标。
- 有序列表、无序列表、任务列表。
- 左对齐、居中、右对齐、两端对齐。
- 字体大小。
- 字体样式。
- 字体颜色。
- 背景高亮色。
- 行高、段前距、段后距。
- 插入链接、图片、附件。
- 插入表格、数据表引用、日历事件引用、待办引用。
- 查找与替换。
- 字数统计。

块级能力：

- 支持拖拽块顺序。
- 支持块菜单。
- 支持快捷命令面板。
- 支持块级评论。
- 支持复制块链接。
- 支持折叠标题下内容。

字体合规要求：

- 当前 UI 中使用 `Hanken Grotesk` 和 `Source Serif 4` 作为无衬线和衬线展示字体。
- 两者均按 SIL Open Font License 1.1 处理，可作为开源项目内置字体候选。
- 若项目实际打包字体文件，必须保留字体版权声明和 OFL 许可证文件。
- 不得内置未经明确授权的商业字体。
- 允许用户自行导入字体，导入时需提示“请确认你拥有该字体在当前使用场景下的合法授权”。
- 用户导入字体仅对其个人或其自托管实例生效，系统应记录字体来源、上传者、上传时间和授权声明确认状态。

验收标准：

- 用户可完成常用 Word 式排版操作。
- 用户导入字体后，可在编辑器字体列表中选择。
- 内置字体许可证信息可在设置或文档中查看。

### 7.4 文档内表格

文档编辑器中的制表能力应参考 Excel 和现代协作文档表格。

基础表格能力：

- 插入普通表格。
- 新增/删除行列。
- 合并/拆分单元格。
- 调整列宽、行高。
- 表头行、表头列。
- 单元格文字对齐。
- 单元格背景色、边框样式。
- 单元格数据类型：文本、数字、日期、勾选、单选、多选、人员、链接、附件。

- 从 Excel/CSV 粘贴识别为表格。

增强表格能力：

- 冻结首行/首列。
- 表格转数据表。

验收标准：

- 从 Excel 复制数据到文档后可保留基础表格结构。
- 表格数据刷新后保持不丢失。

### 7.6 邮箱模块

邮箱模块用于聚合多个邮箱账号，完成邮件收发和基础管理。

功能要求：

- 支持绑定多个邮箱账号。
- 支持 SMTP 发信。
- 支持 IMAP 收信。
- 支持 OAuth2 邮箱授权接入，优先适配 Gmail、Outlook 等标准服务。
- 支持普通邮箱使用授权码或应用专用密码登录。
- 支持收件箱、已发送、草稿、垃圾箱、归档。
- 支持写邮件、回复、转发、删除、标记已读/未读。
- 支持附件上传和下载。
- 支持邮件搜索。
- 支持将邮件转为待办或日历提醒。
- 支持邮件签名。
- 支持邮箱同步状态提示。

手动配置说明：

- 自托管实例必须配置系统发信服务，用于验证码、邀请通知和系统提醒。
- 必填环境变量建议包括 `MAIL_HOST`、`MAIL_PORT`、`MAIL_USER`、`MAIL_PASS`、`MAIL_FROM`、`MAIL_SECURE`。
- 若使用 Gmail、Outlook 等服务，可能需要在服务商后台开启 IMAP/SMTP 或创建应用专用密码。
- 若使用 OAuth2，需在对应服务商后台配置 OAuth 应用、回调地址和客户端密钥。

验收标准：

- 用户可绑定邮箱账号。
- 用户可通过绑定邮箱发送和接收邮件。
- 邮件收发失败时显示可理解的错误原因。

### 7.9 用户系统

用户必须注册登录后才能使用主要功能。

注册登录：

- 邮箱注册。
- 邮箱验证码校验。
- 密码登录。
- 退出登录。
- 忘记密码。
- 重置密码。
- 登录态保持。

账号资料：

- 设置名称。
- 设置头像。
- 修改密码。
- 绑定邮箱。
- 更换邮箱。
- 查看最近登录设备。

邮箱验证码流程：

1. 用户输入邮箱。
2. 前端请求发送验证码。
3. 后端校验邮箱格式、频率限制和风险策略。
4. 后端生成 6 位数字验证码。
5. 验证码哈希后入库，设置过期时间。
6. 系统邮件服务发送验证码。
7. 用户输入验证码。
8. 后端校验验证码、过期时间、尝试次数。
9. 校验成功后完成注册、绑定或换绑。

验证码防刷机制：

- 单邮箱发送间隔不低于 60 秒。
- 单邮箱每小时最多发送 5 次。
- 单 IP 每小时最多发送 20 次。
- 单账号每日最多发送 20 次。
- 验证码有效期 10 分钟。
- 单验证码最多尝试 5 次。
- 验证码必须哈希存储，不得明文存储。
- 发送和验证失败应记录审计日志。
- 触发风险阈值后要求等待、图形验证码或管理员解除。

密码安全：

- 密码长度不少于 8 位。
- 必须服务端哈希存储，推荐 Argon2id 或 bcrypt。
- 登录失败次数过多时临时锁定。
- 修改密码后可选择退出其他设备。

验收标准：

- 用户可完成邮箱验证码注册。
- 用户可登录、退出、修改密码。
- 绑定邮箱和更换邮箱必须经过验证码验证。
- 验证码接口具备频率限制。

### 7.10 分享与协作

分享协作参考腾讯文档、语雀和现代协作文档。

分享方式：

- 通过链接分享。
- 通过指定邮箱邀请用户。
- 通过空间成员或用户组授权。

权限类型：

- 所有者。
- 可管理。
- 可编辑。
- 可评论。
- 可查看。

权限规则：

- 文档所有者可设置默认分享权限。
- 文档所有者可对单个用户设置权限。
- 输入关键词时，可按用户名或邮箱搜索用户。
- 搜索结果需展示头像、名称、邮箱。
- 邮箱匹配用户不存在时，可发送邀请邮件。
- 分享链接可设置有效期。
- 分享链接可设置是否允许匿名访问。
- 分享链接可设置访问密码。
- 分享链接可随时关闭。

实时协作：

- 同一文档多人在线时，显示在线头像和人数。
- 编辑器中显示协作者光标或正在编辑区域。
- 保存状态显示“已保存”“正在同步”“离线编辑”。
- 冲突处理应基于协同编辑协议，不使用最后写入覆盖所有内容的方式。
- 推荐采用 Yjs + WebSocket 实现文档实时协作。

验收标准：

- 所有者可创建分享链接并邀请用户。
- 被邀请用户可按权限查看或编辑文档。
- 所有者可调整或移除他人权限。
- 多人同时编辑时可看到实时在线状态和编辑进度。

### 7.11 管理后台

管理后台用于自托管实例的用户和权限治理。

用户管理：

- 查看用户列表。
- 搜索用户。
- 新增用户。
- 禁用/启用用户。
- 重置用户密码。
- 修改用户角色。
- 查看用户绑定邮箱。
- 查看用户最近登录时间。

用户组管理：

- 创建用户组。
- 编辑用户组名称和描述。
- 添加或移除成员。
- 为用户组配置默认权限。
- 将用户组授权到空间或文档。

系统管理：

- 开放注册开关。
- 邮件服务配置。
- 验证码策略配置。
- 文件上传限制。
- 系统审计日志。
- 数据备份与恢复入口。

验收标准：

- 管理员可登录后台管理用户和用户组。
- 管理员可禁用用户，被禁用用户无法登录。
- 管理员可配置邮件服务并发送测试邮件。

### 7.12 响应式与多设备

项目应支持部署到自有服务器，并通过访问地址在不同设备使用。

断点建议：

- 手机：小于 640px。
- 平板：640px 至 1024px。
- 桌面：1024px 以上。

响应式要求：

- 手机端侧栏收起为抽屉导航。
- 顶栏保留当前页面标题、搜索和用户入口。
- 文档编辑器在手机端优先展示内容，工具栏折叠为底部或浮层。
- 数据表在手机端支持横向滚动和卡片视图。
- 邮箱列表和邮件详情在手机端采用主从页面切换。
- 日历在手机端默认显示日程列表，可切换月视图。
- 管理后台在手机端保持可用，但优先适配桌面管理场景。

验收标准：

- 手机、平板、桌面均可完成登录、阅读、编辑、查看任务。
- 页面不出现核心操作被遮挡或不可点击。
- 多设备访问同一账号时数据保持同步。

## 8. 推荐技术方案

### 8.1 前端

- 沿用 React + TypeScript + Vite。
- 沿用 Tailwind CSS 和当前设计 token。
- 引入 React Router 实现页面路由。
- 编辑器推荐 TipTap 或 Lexical。
- 数据表可先自研轻量数据模型，后续再抽象高级公式和视图。
- 状态管理推荐 Zustand 或 React Query + 本地 state 组合。

### 8.2 后端

- 推荐 Node.js + Express。
- 当前项目依赖已包含 `express` 和 `dotenv`，可在后续实现中补齐服务端入口。
- API 风格推荐 REST，实时协作和通知使用 WebSocket。
- 邮箱、验证码、文件上传、协作同步应在后端完成，不应暴露敏感密钥到前端。

### 8.3 数据库

- 生产部署推荐 PostgreSQL。
- 本地轻量部署可提供 SQLite 适配。
- 推荐使用 Prisma 或 Drizzle 作为 ORM。
- 数据库迁移文件应纳入版本管理。

### 8.4 实时协作

- 推荐 Yjs + WebSocket。
- 文档内容以协同数据结构存储。
- 需要保存定期快照，支持版本恢复。

### 8.5 邮件服务

- 系统发信：SMTP。
- 用户邮箱聚合：IMAP + SMTP，OAuth2 作为高级接入方式。
- 邮箱密码或授权码必须加密存储。
- 推荐为每个用户邮箱账号保存同步游标，避免重复拉取。

## 9. 数据模型建议

核心实体：

- User：用户。
- UserProfile：用户资料。
- UserGroup：用户组。
- Session：登录会话。
- EmailVerificationCode：邮箱验证码。
- MailAccount：用户绑定邮箱。
- MailMessage：邮件索引。
- Workspace：空间。
- Document：文档。
- DocumentVersion：文档版本。
- DocumentPermission：文档权限。
- ShareLink：分享链接。
- TableBase：数据表。
- TableField：字段。
- TableRecord：记录。
- CalendarEvent：日程。
- TodoItem：待办。
- AuditLog：审计日志。
- SystemSetting：系统设置。

权限判断顺序：

1. 系统管理员权限。
2. 文档所有者权限。
3. 直接用户权限。
4. 用户组权限。
5. 分享链接权限。
6. 空间默认权限。
7. 无权限。

## 10. 安全与合规

- 所有需要登录的 API 必须校验 session 或 token。
- 用户只能访问自己有权限的文档、表格、邮件、日程和待办。
- 邮件授权信息、OAuth token、SMTP 密码必须加密存储。
- 验证码不得明文存储。
- 文件上传需限制大小、类型和访问权限。
- 管理后台操作需记录审计日志。
- 分享链接 token 应使用高强度随机值。
- 支持关闭公开注册。
- 开源项目不得默认内置未经授权的商业字体、图标或素材。

## 11. 部署与配置

推荐部署方式：

- 单机 Docker Compose：前端、后端、数据库、文件存储。
- 手动部署：Node.js 服务 + PostgreSQL + Nginx 反向代理。
- 本地开发：Vite dev server + Express dev server + SQLite 或 PostgreSQL。

关键环境变量：

```text
APP_URL=
DATABASE_URL=
SESSION_SECRET=
MAIL_HOST=
MAIL_PORT=
MAIL_USER=
MAIL_PASS=
MAIL_FROM=
MAIL_SECURE=
UPLOAD_MAX_SIZE=
ALLOW_PUBLIC_REGISTER=
RATE_LIMIT_WINDOW_SECONDS=
RATE_LIMIT_MAX_REQUESTS=
```

邮箱验证码手动配置说明：

- 必须配置可用 SMTP 服务，否则注册、绑定邮箱、换绑邮箱、邀请邮件无法发送。
- 个人邮箱服务通常需要开启 SMTP 并使用应用专用密码。
- 企业邮箱需要管理员在邮箱后台开启 SMTP/IMAP 权限。
- 生产环境必须配置正确的 `APP_URL`，用于生成邀请链接和回调地址。
- 若服务器云厂商默认封锁 25 端口，应改用 465、587 或邮件服务商 API。

## 16. 风险与注意事项

- 实时协作、邮箱聚合和数据表是复杂度最高的模块，应分阶段实现。
- 邮件收发涉及不同服务商策略，需提供清晰的手动配置文档和错误提示。
- 用户导入字体可能存在版权风险，系统必须保留授权确认提示。
- 自托管环境差异较大，部署文档需覆盖端口、HTTPS、邮件、数据库和备份。
- 当前工程仍是静态 UI 原型，后续实现应优先建立路由、数据模型、API 和测试基础，再逐步补齐高级功能。
