# 简记配置说明

本文档说明自托管部署和首次网页初始化需要配置的内容。请不要把真实 `.env`、SMTP 密码、JWT 密钥、初始化链接提交到开源仓库。

## 1. 后端环境变量

配置文件路径：

- 本地开发：`server/.env`
- 配置模板：`server/.env.example`
- Docker 部署：项目根目录 `.env` 或部署平台环境变量，供 `docker-compose.yml` 读取

Docker 生产部署必填项：

```env
NODE_ENV=production
PORT=4000
APP_URL=https://你的域名

DATABASE_URL=file:/app/data/dev.db

JWT_SECRET=至少32位的随机字符串
JWT_EXPIRES_IN=7d
COOKIE_NAME=jianji_session
COOKIE_SECURE=true
SETUP_TOKEN=至少32位的随机字符串

# 网页初始化完成前保持 false；SMTP 会在初始化页面填写并加密保存到数据库。
MAIL_ENABLED=false
ALLOW_PUBLIC_REGISTER=true

# 可选：版本展示与后台更新按钮
APP_VERSION=0.1.0
JIANJI_LATEST_VERSION=
JIANJI_CURRENT_COMMIT=
JIANJI_UPDATE_REPO=https://github.com/staklab/jianji.git
JIANJI_UPDATE_BRANCH=main
JIANJI_UPDATE_CHECK_URL=https://api.github.com/repos/staklab/jianji/commits/main
JIANJI_UPDATE_ARCHIVE_URL=
JIANJI_UPDATE_COMMAND=
```

注意：

- `JWT_SECRET` 会用于登录会话和邮箱聚合凭据加密，部署后不要随意更换。
- `SETUP_TOKEN` 用于首次配置页面鉴权。推荐由 `scripts/install.sh` 自动生成，不要公开。
- 使用 HTTPS 域名时建议设置 `COOKIE_SECURE=true`。
- `APP_URL` 要写用户实际访问地址，例如 `https://jianji.example.com`，不要在生产环境继续使用 `localhost`。
- 管理员邮箱、管理员密码、SMTP 授权码不需要写入生产 `.env`，请在首次访问私密初始化链接时填写。
- `APP_VERSION` 用于后台展示当前版本；`JIANJI_LATEST_VERSION` 可由部署者或发布脚本写入最新版本号。
- `JIANJI_CURRENT_COMMIT`、`JIANJI_UPDATE_REPO`、`JIANJI_UPDATE_BRANCH` 会由安装和更新脚本自动写入，用于判断远端 `main` 分支是否有新提交。
- `JIANJI_UPDATE_CHECK_URL` 用于检测最新版本，默认读取 GitHub main 分支最新提交；不需要 DNS API 或云解析权限。
- `JIANJI_UPDATE_ARCHIVE_URL` 用于非 Git checkout 部署下载源码归档；为空时默认使用 GitHub 分支归档。若服务器无法直连 GitHub，可改为你信任的镜像或自建反代地址。
- `JIANJI_UPDATE_COMMAND` 为空时，后台【版本更新】只负责通知用户并提示管理员手动 SSH 更新；如需后台按钮自动执行，可设置为 `bash /opt/jianji/scripts/update.sh`。

## 2. 首次网页初始化

一键部署脚本完成后会输出私密链接：

```text
https://你的域名/setup?token=SETUP_TOKEN
```

脚本也会把该链接保存到服务器部署目录的 `SETUP_URL.txt`。例如部署在 `/opt/jianji` 时可通过以下命令查看：

```bash
cat /opt/jianji/SETUP_URL.txt
```

请由部署管理员打开该链接并完成：

- 确认站点访问地址。
- 创建管理员邮箱、显示名称和密码。
- 配置 SMTP 服务器、端口、SSL/TLS、用户名、授权码/应用密码、发信人。
- 选择是否允许公开注册。

安全逻辑：

- 没有任何用户时才允许初始化。
- 普通访客访问未初始化实例时只会看到锁定提示，不会看到配置表单。
- 初始化密钥只换取短时 HttpOnly 会话，前端会移除 URL 中的 token。
- SMTP 授权码会使用 `JWT_SECRET` 派生密钥加密后写入后端数据库的 `SystemSetting.mail_pass_enc`。
- 初始化完成后配置表单自动关闭。

## 3. SMTP 邮箱验证码

注册、换绑邮箱、找回密码验证码都通过 SMTP 发送。非测试环境下如果 SMTP 配置缺失，验证码接口会返回错误，不会再把验证码打印到终端。

常见服务商配置要点：

- QQ、网易、企业邮箱通常需要开启 SMTP 并使用授权码，不要填写网页登录密码。
- Gmail 需要开启两步验证后创建应用专用密码。
- Outlook / Microsoft 365 可能需要管理员开启 SMTP AUTH。
- 云服务器常封锁 25 端口，优先使用 465 或 587。

初始化页面会先验证 SMTP 可连接再保存。配置完成后，可在【管理后台】-【系统设置】-【发送测试邮件】中做冒烟测试。

## 4. Docker 部署

`docker-compose.yml` 会读取部署环境变量。推荐在项目根目录创建只保存在服务器上的 `.env`：

```env
APP_URL=https://jianji.example.com
JWT_SECRET=至少32位的随机字符串
SETUP_TOKEN=至少32位的随机字符串
COOKIE_SECURE=true
MAIL_ENABLED=false
APP_VERSION=0.1.0
JIANJI_LATEST_VERSION=
JIANJI_CURRENT_COMMIT=
JIANJI_UPDATE_REPO=https://github.com/staklab/jianji.git
JIANJI_UPDATE_BRANCH=main
JIANJI_UPDATE_CHECK_URL=https://api.github.com/repos/staklab/jianji/commits/main
JIANJI_UPDATE_ARCHIVE_URL=
JIANJI_UPDATE_COMMAND=
```

启动：

```bash
docker compose up -d --build
```

持久化数据卷：

- `jianji-data`：SQLite 数据库，容器内路径 `/app/data`
- `jianji-uploads`：上传附件、头像等文件

Docker 部署时推荐使用：

```env
DATABASE_URL=file:/app/data/dev.db
```

不要把数据卷挂载到 `/app/prisma`，否则会覆盖镜像内置的 Prisma schema 与 migrations。

## 5. 更新、备份与迁移

推荐更新命令：

```bash
bash scripts/update.sh
```

该脚本会：

- 备份 `.env` 到 `.update-backups/.env.时间戳`。
- 备份 `SETUP_URL.txt` 到 `.update-backups/SETUP_URL.txt.时间戳`。
- 如果部署目录是 Git 仓库，则执行 `git fetch --all --prune` 和 `git pull --ff-only`；如果不是 Git 仓库，则从 `JIANJI_UPDATE_REPO` / `JIANJI_UPDATE_BRANCH` 指向的 GitHub 分支归档下载源码，并保护 `.env`、初始化链接、证书、数据库、上传文件和备份目录。
- 刷新 `APP_VERSION`、`JIANJI_CURRENT_COMMIT`、`JIANJI_UPDATE_CHECK_URL` 等部署元数据。
- 执行 `docker compose up -d --build` 重建并重启服务。
- 等待 Docker 健康检查通过后再宣布更新完成；若失败会输出服务状态和尾部日志。
- 不删除 SQLite 数据卷和上传文件卷。

如果服务器使用宿主机构建、`jianji:runtime` 镜像运行的低内存优化部署，请改用：

```bash
bash scripts/update-runtime.sh
```

该脚本会在宿主机完成依赖安装、Prisma Client 生成、前后端构建，然后重新打包 `jianji:runtime` 镜像并用 `docker-compose.runtime.yml` 重启容器，仍会保留现有数据卷。

当前更新机制不强制依赖 GitHub Releases：后台默认通过 GitHub `main` 分支最新 commit 判断是否有新版本，因此只要仓库推送到 `main`，服务器就能检测到更新。Releases 更适合作为稳定里程碑、Release Notes 和回滚说明；如果只是自托管 Web 应用持续迭代，commit 更新已经足够支撑无损更新。

如果部署环境无法稳定访问 GitHub 的仓库接口或源码归档地址，服务器端自动拉取更新将不可用，可选择以下方式：

- 在服务器系统或服务环境中配置可信 HTTP/HTTPS 代理，让 `curl`、`git` 和后台版本检查都能访问 GitHub。
- 自建可信反代/镜像，把 `JIANJI_UPDATE_CHECK_URL` 指向返回 `{"sha":"..."}` 的检查地址，把 `JIANJI_UPDATE_ARCHIVE_URL` 指向 `main.tar.gz` 源码归档地址。
- 在能获取最新源码的本地电脑上执行推送式更新：

```bash
git pull
bash scripts/push-update.sh --host root@example.com --dir /opt/jianji --runtime
```

推送式更新使用 SSH/rsync 同步源码，并在服务器上以 `JIANJI_SKIP_SOURCE_REFRESH=true` 执行更新脚本；服务器 `.env`、`SETUP_URL.txt`、证书、数据库、上传文件和备份目录都会被保留。

后台【系统设置】-【版本更新】支持手动和自动两种模式：

- 未配置 `JIANJI_UPDATE_COMMAND`：点击更新时先给用户发送站内通知和邮件，然后提示管理员 SSH 执行更新脚本。
- 已配置 `JIANJI_UPDATE_COMMAND`：后台会执行该命令，更新开始和完成都会给用户发送站内通知和邮件。

后台【系统设置】-【备份与恢复】支持：

- 数据库备份：导出和恢复核心业务数据。
- 完整迁移包：导出数据库、上传文件和脱敏后的配置摘要，适合换服务器迁移。

迁移包可能包含加密后的邮箱凭据和用户上传文件，应作为私密备份保存。迁移包不会包含明文 `.env`、`SETUP_TOKEN`、`JWT_SECRET`、SMTP 明文密码、DNS API Key、证书私钥等部署机密。

## 6. Nginx 反向代理

生产环境推荐让 Nginx 代理到后端容器的 `4000` 端口。前端 API 使用相对路径 `/api`，因此部署在域名下时会自动走当前域名，不需要保留 `localhost`。

示例：

```nginx
server {
  listen 443 ssl http2;
  server_name jianji.example.com;

  location / {
    proxy_pass http://127.0.0.1:4000;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  }
}
```

## 7. 生产排障命令

服务状态：

```bash
docker compose ps
docker compose logs -f jianji
curl -fsS https://你的域名/api/health
```

确认是否 SSH 到正确服务器：

```bash
hostname
hostname -I
nproc
free -h
```

查看首次初始化链接：

```bash
cat /opt/jianji/SETUP_URL.txt
```

如果使用 Nginx 和 HTTPS，还可检查：

```bash
nginx -t
systemctl status nginx
```

## 8. 安全检查清单

- 不提交真实 `server/.env`。
- 不提交生产 `.env`、`SETUP_URL.txt`、数据库文件、上传文件、证书、DNS API Key。
- 不公开首次初始化链接或 `SETUP_TOKEN`。
- 生产环境管理员账号只在首次网页初始化中创建。
- 生产环境必须设置强 `JWT_SECRET`。
- 首次初始化必须配置并验证 SMTP，避免注册和找回密码不可用。
- 确认 `APP_URL`、Nginx 域名和浏览器访问域名一致。
- 备份 SQLite 数据卷和 `uploads` 数据卷。
